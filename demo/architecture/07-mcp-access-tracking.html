Demo 07 MCP Access Tracking

← Prev Index Next →

Demonstrates the full integration: governance allow + deny, a live MCP tool call with OAuth authentication, access recording in the database, and a complete audit trail. Cost: free (direct API calls + MCP calls, no AI).

1 Governance — ALLOW
$ curl -s -X POST "http://localhost:8080/api/public/hooks/govern?plugin_id=enterprise-demo" ... -d '{"tool_name":"Read","agent_id":"developer_agent","tool_input":{"file_path":"/src/main.rs"}}'
What happens Clean Read tool input, all rules pass.
Output ALLOW
Why Baseline verification — before testing denial, confirm the allow path works. This establishes that the rule engine is functioning and clean inputs pass through.
2 Governance — DENY
$ curl -s -X POST "http://localhost:8080/api/public/hooks/govern?plugin_id=enterprise-demo" ... -d '{"tool_name":"Bash","agent_id":"developer_agent","tool_input":{"command":"curl -H \"Authorization: AKIAIOSFODNN7EXAMPLE\" ..."}}'
What happens scope_check passes (admin), but secret_injection detects AWS key pattern → DENY.
Output DENY
Why Independent rule enforcement — scope doesn't override security. Even though the agent has admin scope, the secret_injection rule blocks the call. Rules are evaluated independently.
3 MCP Tool Call — Authenticated
$ plugins mcp call skill-manager list_plugins --profile local
What happens OAuth client credentials flow → access token → tool execution → result returned → access recorded in user_activity table (category='mcp_access').
Sub-flow
3aOAuth — Client credentials → access token validated against MCP server
3bExecution — skill-manager.list_plugins() → JSON plugin list
3cRecording — record_mcp_access() → INSERT INTO user_activity
Why Typed OAuth flow — the OAuth exchange returns typed TokenResponse { access_token, token_type, expires_in }, not a raw JSON blob. record_mcp_access() takes typed McpServerId and UserId — you can't accidentally log the wrong server or user.
4 Audit Trail — Database
$ infra db query "SELECT decision, tool_name, reason FROM governance_decisions ORDER BY created_at DESC LIMIT 5"
$ infra db query "SELECT action, entity_name, description FROM user_activity WHERE category = 'mcp_access' ORDER BY created_at DESC LIMIT 5"
What happens Queries two tables — governance_decisions shows allow/deny from Parts 1–2, user_activity shows MCP access events from Part 3.
Why Dual audit tables — governance decisions and MCP access are stored in separate tables with different schemas because they serve different purposes. governance_decisions tracks policy enforcement. user_activity tracks usage patterns. Both use compile-time checked SQL (sqlx::query!).
5 Dashboard
URL http://localhost:8080/admin/
What appears
•Policy Violations section — governance denials
•MCP Server Access section — auth events + tool calls
•Governance Decisions section — allow/deny with reasons
Why Single pane of glass — all events from all surfaces (CLI, Claude Code hooks, direct API) appear in one dashboard, linked by the same UserId from the JWT.
