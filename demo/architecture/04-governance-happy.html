Demo 04 Governance: Allow

← Prev Index Next →

Direct call to the governance API simulating a Claude Code PreToolUse hook. Admin-scope agent with clean tool input — all 3 rules pass. Uses Read tool (not MCP) to exercise scope_check, secret_injection, and rate_limit. Cost: free.

1 Agent Calls MCP Tool
$ admin agents message developer_agent -m "List all agents" --context-id $CONTEXT_ID --blocking
What happens developer_agent (admin scope) decides to call mcp__systemprompt__list_agents. Claude Code fires the PreToolUse hook synchronously before tool execution.
Why Synchronous enforcement — the hook blocks execution. The tool cannot run until governance returns allow or deny. This is the single enforcement point in the entire pipeline.
2 HTTP POST to Governance
POST /api/public/hooks/govern?plugin_id=enterprise-demo
What happens Hook sends JSON payload with hook_event_name, tool_name, agent_id, session_id, tool_input. Authorization: Bearer JWT.
Detail JSON enters at the HTTP boundary (the only untyped moment). Axum handler deserializes via serde into HookEventPayload. From here, all data is typed Rust structs.
Why Typed boundary — JSON is untyped only at the HTTP edge. serde::from_value immediately converts to validated Rust structs. GovernanceContext carries UserId, SessionId, AgentName as distinct newtypes.
3 Rule Engine Evaluation ~12ms
What happens Three rules evaluated in order:
3ascope_check — admin scope → all tools allowed → PASS
3bsecret_injection — scan tool_input for AWS keys, PATs, PEM → PASS (clean)
3crate_limit — <300 calls/min this session → PASS
Decision All PASS → decision = "allow", policy = "default_allow"
Why Defense in depth — three independent rules. Even if scope allows access, secret injection still blocks leaked credentials. Rate limiting prevents runaway agents. Rules are evaluated as Vec<RuleEvaluation> — typed structs, not loose JSON.
4 Audit + Response
(internal) tokio::spawn(record_decision())
What happens Async INSERT INTO governance_decisions with compile-time checked SQL (sqlx::query!). Response: HTTP 200 with { permissionDecision: "allow" }. Claude Code receives "allow" → tool executes.
Why Non-blocking audit — tokio::spawn fires the DB write without blocking the response. The governance decision returns immediately (~12ms) while the audit record writes asynchronously. This keeps the hot path fast.
