Demo 08 Request Tracing

← Prev Index

End-to-end request tracing: typed data flow, all newtype IDs, CLI log commands, system pipeline, and a 200-request benchmark. Cost: free (governance API calls + DB queries, no AI). This is the deep-dive into how every request flows through the typed Rust pipeline.

1 Typed Data Flow
$ curl -s -X POST ... -d '{"hook_event_name":"PreToolUse","tool_name":"Read","agent_id":"developer_agent","session_id":"demo-trace-...","tool_input":{"file_path":"/src/main.rs"}}'
What happens Governance request + track request, shows full payloads. JSON enters at HTTP boundary, immediately deserialized into Rust structs.
Sub-flow
1aGovernance — PreToolUse → allow/deny, captures HTTP status + response time
1bTrack — PostToolUse → event recorded, captures HTTP status + response time
Field types hook_event_name: String  •  tool_name: String  •  agent_id: String  •  session_id → SessionId (newtype)  •  tool_input → serde_json::Value  •  permissionDecision: "allow"|"deny"
Why Single untyped boundary — JSON is untyped ONLY at the HTTP edge. serde deserializes into HookEventPayload immediately. From that point, session_id becomes SessionId (newtype) — the compiler prevents passing it where a UserId is expected. Zero runtime cost for this type safety.
2 All Typed IDs
$ infra db query "SELECT id, user_id, session_id, agent_id, decision, plugin_id FROM governance_decisions WHERE session_id = '$SESSION_ID' ORDER BY created_at DESC LIMIT 3"
IDs in system decision_id UUID PK  •  user_id JWT sub  •  session_id client payload  •  agent_id hook payload  •  plugin_id query param  •  trace_id per-request  •  context_id CLI created
Rust newtypes pub struct UserId(String);    pub struct SessionId(String);
Why Compile-time ID safety — fn record(user: &UserId, session: &SessionId) — the compiler enforces correct types at every call site. You cannot pass a SessionId where a UserId is expected. Every SQL query that writes these uses sqlx::query! — checked against the live schema at compile time.
3 Log Commands
Trace list infra logs trace list --limit 3
execution_traces table — TraceId, AgentName, status, event count
Trace detail infra logs trace show <id> --all
Trace events — AI requests, tool calls, execution steps, skills, governance
AI requests infra logs request list --limit 5
ai_requests table — model, tokens, cost per request
App logs infra logs view --level info --since 5m
Application logs — tracing crate structured output
Why Observable by default — every request is traced. No special debug mode needed. The CLI provides 4 different views into the same data, from summary (trace list) to full detail (trace show --all).
4 Request Flow Map
Pipeline
Client → Router HookEventPayload via serde → JWT JwtClaims { sub: UserId } → Scope GovernanceContext → Rules Vec<RuleEvaluation> → Audit sqlx::query! INSERT → Response GovernanceResponse
Why Typed at every stage — no dictionaries, no raw strings, no unvalidated JSON passes between stages. If you change a column name in a migration, every query that touches it fails to compile.
