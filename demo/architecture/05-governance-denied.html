Demo 05 Governance: Deny

← Prev Index Next →

Two denial scenarios via direct governance API calls. Part 1: scope restriction — user-scope agent blocked from admin tools. Part 2: blocklist — destructive tool blocked regardless of scope. Cost: free.

1 Direct API Call Part 1 of demo
$ curl -s -X POST "http://localhost:8080/api/public/hooks/govern?plugin_id=enterprise-demo" -H "Authorization: Bearer $TOKEN" -d '{"hook_event_name":"PreToolUse","tool_name":"mcp__systemprompt__list_agents","agent_id":"associate_agent","session_id":"demo-governance-denied"}'
What happens Direct API call to governance endpoint. tool_name starts with mcp__systemprompt__ (admin-only). agent_id is associate_agent (user scope).
Output { permissionDecision: "deny", permissionDecisionReason: "[GOVERNANCE] user scope cannot access admin-only tools" }
Why Defense at the rule level — Demo 02 denies at the mapping level (no tools available). This demo denies at the rule level — the agent HAS MCP tools, but governance blocks the specific tool call. Two layers of defense.
2 Live Agent Attempt Part 2
$ admin agents message associate_agent -m "List all agents using CLI tools" --context-id $CONTEXT_ID --blocking
What happens associate_agent tries to call MCP tool. PreToolUse hook fires. Governance returns DENY. Claude Code blocks the tool. Agent reports "I cannot execute this operation."
Why Deterministic blocking — the permissionDecision is not a suggestion. Claude Code enforces it. The agent cannot override a deny.
3 Rule Engine internal
scope_check tool requires admin, agent is user → FAIL
secret_injection skipped (first failure is decisive)
rate_limit skipped
Decision deny — policy = scope_restriction
Why Short-circuit evaluation — first failure stops evaluation. No point checking secrets or rate limits if scope already denies. Keeps governance fast.
4 Audit
What happens INSERT INTO governance_decisions with decision="deny", policy="scope_restriction", evaluated_rules showing scope_check fail.
Why Audit completeness — denials are audited with the same fidelity as approvals. The evaluated_rules JSONB stores exactly which rule failed and why.
