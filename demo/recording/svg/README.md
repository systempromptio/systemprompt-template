# Terminal SVG recordings

These are the razor-sharp terminal SVGs embedded in the project `README.md`. They are generated by running the `svg-{pillar}-*.sh` demo scripts under [asciinema](https://asciinema.org/) and rendering each cast with [svg-term-cli](https://github.com/marionebl/svg-term-cli).

Scripts are organized by the three brochure pillars: `infra-*` (Infrastructure), `cap-*` (Capabilities), `int-*` (Integrations). Each SVG maps to a feature page on [systemprompt.io](https://systemprompt.io).

## Layout

```
svg/
├── record.sh                      # Orchestrator — regenerates all recordings
├── _colors.sh                     # Shared helpers (header, type_cmd, pass/fail/check, …)
├── profiles/
│   ├── dark.xrdb                  # Dark palette (README dark mode)
│   └── light.xrdb                 # Light palette (README light mode)
├── svg-infra-self-hosted.sh       # Infrastructure pillar
├── svg-infra-deploy-anywhere.sh
├── svg-infra-control-plane.sh
├── svg-infra-open-standards.sh
├── svg-cap-governance.sh          # Capabilities pillar
├── svg-cap-secrets.sh
├── svg-cap-mcp.sh
├── svg-cap-analytics.sh
├── svg-cap-agents.sh
├── svg-cap-compliance.sh
├── svg-int-any-agent.sh           # Integrations pillar
├── svg-int-extensions.sh
├── svg-int-benchmark.sh
├── recordings/                    # asciinema .cast files (regenerated, committed)
└── output/
    ├── dark/*.svg                 # Dark-mode SVGs embedded in README
    └── light/*.svg                # Light-mode SVGs embedded in README
```

## Regenerate

```bash
just start                                         # server must be up
./demo/00-preflight.sh                             # seed + write demo/.token
just record-svgs                                   # regenerate all
just record-svgs governance secrets                # regenerate matching scenes only
just record-svgs infra                             # all infrastructure scenes
```

`just record-svgs` shells out to `demo/recording/svg/record.sh`, which for every `svg-{infra,cap,int}-*.sh`:

1. Runs the script under `asciinema rec` → `recordings/{name}.cast`
2. Renders a dark SVG → `output/dark/{name}.svg`
3. Renders a light SVG → `output/light/{name}.svg`

## Prerequisites

| Tool | Install |
|---|---|
| `asciinema` | `pipx install asciinema` or `brew install asciinema` |
| `svg-term` | `npm i -g svg-term-cli` |
| Live server | `just start` (port 8080) |
| Auth token | `./demo/00-preflight.sh` (writes `demo/.token`) |

## Add a new recording

1. Copy an existing script, e.g. `cp svg-cap-governance.sh svg-cap-your-topic.sh`. Use the pillar prefix: `infra-`, `cap-`, or `int-`.
2. Source `_colors.sh`; use `header`, `subheader`, `type_cmd`, `pass`/`fail`/`check`, `divider`. Keep it under ~40 seconds of real-time playback so the SVG stays embed-friendly.
3. Hit real endpoints — no mocks. Each recording doubles as an integration smoke test.
4. Run `just record-svgs your-topic` to capture only the new one.
5. Commit the `svg-{pillar}-*.sh`, the `recordings/*.cast`, and the two `output/**/*.svg` files together. Reference the SVG from `README.md` with a `<picture>` block so it respects dark/light mode.

## Conventions

- **Width:** 100 cols, **Height:** 32 rows. Anything wider wraps in the README column.
- **Colors:** only the palette defined in `_colors.sh`. svg-term renders anything else as default text.
- **Commands via `type_cmd`** (not raw `echo`) — the keystroke animation is what makes the recording feel alive.
- **Real data only.** Every cURL call, every CLI command runs for real against the local server. When demos drift, these scripts break loudly and visibly.
