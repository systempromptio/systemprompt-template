# Hooks

**TL;DR:** Hooks are event-driven triggers that execute commands when specific events occur in the platform. They let you automate actions like tracking tool usage, logging session activity, or sending notifications -- without modifying agent code. Hooks come in two varieties: **system hooks** that are auto-generated for platform-wide tracking, and **custom hooks** that you create for your own automation needs.

## What You'll See

Navigate to **Hooks** in the admin sidebar. The page has three distinct sections:

### Metric Ribbon

At the top of the page, a horizontal ribbon displays four key metrics at a glance:

| Metric | What it means |
|--------|---------------|
| **System Hooks** | Number of distinct system event types being tracked |
| **Tracked Plugins** | Number of plugins that have system hooks attached |
| **Custom Hooks** | Number of custom hooks you have created |
| **Total Hooks** | Combined count of system and custom hooks |

### Toolbar

Below the ribbon, a toolbar provides a search box to filter hooks by name and a **+ New Hook** button to create a custom hook.

### Hooks Table

A unified data table lists all hooks -- system hooks first, then custom hooks. The columns are:

| Column | What it shows |
|--------|---------------|
| **Name** | The hook's display name, with a description underneath (if set) |
| **Event** | The event type that triggers this hook (shown as inline code) |
| **Command** | The command that runs when the hook fires (truncated, with full text on hover) |
| **Plugins** | Badge(s) showing which plugin(s) this hook is associated with |
| **Type** | A badge indicating **System**, **Async**, or **Sync** |
| **Actions** | Edit and Delete options (only shown for custom hooks -- system hooks cannot be edited or deleted) |

### RBAC Visibility

Non-admin users only see hooks belonging to plugins assigned to their roles. Hooks with a `visible_to` restriction are further filtered -- only users with a matching role (or admins) can see them.

## System Hooks vs. Custom Hooks

### System Hooks

System hooks are auto-generated by the platform to track core events across plugins. They appear in the table with a **System** badge and cannot be edited or deleted from the dashboard.

System hooks are deduplicated by event type. If multiple plugins define the same system event, the hooks table shows a single row with multiple plugin badges.

The platform defines these system hook types:

| Event | System Hook Name | Description |
|-------|-----------------|-------------|
| `PostToolUse` | Usage Tracking | Records tool usage analytics after each tool call |
| `PostToolUseFailure` | Failure Tracking | Records tool failure events for error analytics |
| `UserPromptSubmit` | Prompt Tracking | Tracks user prompt submissions |
| `SessionStart` | Session Start Tracking | Records when user sessions begin |
| `SessionEnd` | Session End Tracking | Records when user sessions end |
| `Stop` | Stop Event Tracking | Tracks agent stop events |
| `SubagentStart` | Subagent Start Tracking | Records when subagents are spawned |
| `SubagentStop` | Subagent Stop Tracking | Records when subagents complete |

### Custom Hooks

Custom hooks are ones you create to automate your own workflows. They appear after system hooks in the table and can be edited and deleted.

## Creating a Custom Hook

1. Click **+ New Hook** in the toolbar (or navigate to `/admin/hooks/edit/`).
2. Fill in the form:

| Field | Required | Description |
|-------|----------|-------------|
| **Name** | Yes | A descriptive name (e.g., "Sound bell hook"). If left empty, the command is used as the display name. |
| **Description** | No | What this hook does (text area, 2 rows). |
| **Plugin** | Yes | Select which plugin this hook belongs to. The dropdown shows plugins available to your roles. |
| **Event** | Yes | The event type that triggers this hook (see Event Types below). |
| **Matcher** | No | A pattern to filter which specific events trigger the hook. |
| **Command** | No | The shell command or script to execute when the hook fires (text area, 3 rows). |
| **Run Async** | -- | Checkbox. When checked, the command runs in the background without blocking. |

3. Click **Save** to create the hook.

## Event Types

The following event types are available when creating a hook:

| Event | When It Fires |
|-------|---------------|
| `PostToolUse` | After an MCP tool is successfully invoked |
| `PostToolUseFailure` | After an MCP tool invocation fails |
| `PreToolUse` | Before an MCP tool is invoked (can intercept) |
| `UserPromptSubmit` | When a user submits a prompt |
| `SessionStart` | When a new user session begins |
| `SessionEnd` | When a user session ends |
| `Stop` | When an agent stops processing |
| `SubagentStart` | When a subagent is spawned |
| `SubagentStop` | When a subagent completes |
| `Notification` | When a notification event occurs |

## Matchers

The matcher field filters which specific events trigger the hook. How the matcher is evaluated depends on the event type:

- For `PostToolUse` / `PostToolUseFailure` / `PreToolUse` -- matched against the tool name
- For `SessionStart` / `SessionEnd` -- matched against the session context
- An **empty matcher** matches all events of that type

Examples:
- `"Skill"` -- fires only when a tool with "Skill" in the name is used
- `"search"` -- fires only for tools or contexts containing "search"
- `""` (empty) -- fires on every event of the selected type

## Synchronous vs. Asynchronous

| Mode | Checkbox | Behavior |
|------|----------|----------|
| **Synchronous** | Unchecked | The platform waits for the command to finish before continuing. Use when the hook's result affects the next step. |
| **Asynchronous** | Checked | The command runs in the background. The platform continues immediately. Use for analytics, logging, and notifications. |

The hooks table shows a **Sync** or **Async** badge for each custom hook so you can see the execution mode at a glance.

## Editing a Custom Hook

1. Click the action menu on any custom hook row and select **Edit**.
2. Modify any field.
3. Click **Save**.

System hooks cannot be edited.

## Deleting a Custom Hook

1. Click the action menu on the hook row.
2. Select **Delete**.
3. Confirm the deletion.

Deletion requires admin access. System hooks cannot be deleted.

## Hook Configuration in Plugin YAML

Hooks are defined within a plugin's `config.yaml`. When you create a hook through the dashboard, it is stored in the selected plugin's configuration:

```yaml
# services/plugins/your-plugin/config.yaml
plugin:
  hooks:
    PostToolUse:
      - matcher: "Skill"
        hooks:
          - type: command
            command: "${CLAUDE_PLUGIN_ROOT}/scripts/track-usage.sh"
            async: true

    SessionStart:
      - matcher: ""
        hooks:
          - type: command
            command: "${CLAUDE_PLUGIN_ROOT}/scripts/init-session.sh"
            async: true
```

### Environment Variables

Hook commands have access to these environment variables:

| Variable | Description |
|----------|-------------|
| `CLAUDE_PLUGIN_ROOT` | Absolute path to the plugin's directory |

## Example: Usage Analytics

Track every tool invocation for analytics:

```yaml
plugin:
  hooks:
    PostToolUse:
      - matcher: ""
        hooks:
          - type: command
            command: "${CLAUDE_PLUGIN_ROOT}/scripts/track-usage.sh"
            async: true
    PostToolUseFailure:
      - matcher: ""
        hooks:
          - type: command
            command: "${CLAUDE_PLUGIN_ROOT}/scripts/track-failure.sh"
            async: true
```

Both hooks run asynchronously so they do not slow down agent responses.

## How Hooks Connect to Other Concepts

- **Plugins** own hooks. Every hook belongs to a plugin, and the plugin dropdown on the create form determines where the hook is stored.
- **Agents** generate the events that hooks respond to -- session starts, tool usage, prompt submissions.
- **MCP Servers** provide the tools whose invocations trigger `PostToolUse`, `PostToolUseFailure`, and `PreToolUse` events.
- **Skills** are often the specific tools being tracked through `PostToolUse` hooks with a `"Skill"` matcher.

## Troubleshooting

**Hook not firing** -- Verify the event type is correct and the matcher matches the expected value. Check that the plugin the hook belongs to is enabled. An empty matcher matches all events, so try clearing it to test.

**Command fails silently** -- Check that the script exists at the specified path and has execute permissions. For async hooks, errors are logged but do not surface to the user. Check `systemprompt infra logs view --level error --since 1h`.

**Cannot edit or delete a hook** -- System hooks are read-only and cannot be modified from the dashboard. If you need to change system hook behavior, modify the plugin's YAML configuration directly.

**Hook not visible** -- Hooks with `visible_to` restrictions are only shown to users with matching roles. Ask an admin to check the hook's visibility settings.

**Async hook not completing** -- Async hooks run in the background with no timeout guarantee. Ensure scripts handle their own error cases and do not hang. Check system logs for crash output.
