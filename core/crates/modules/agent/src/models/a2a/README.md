# A2A Protocol Domain Models

This module contains ALL type definitions from the A2A (Agent-to-Agent) protocol specification.
These are pure data structures with no business logic - just serialization/deserialization.

## Purpose
- Define all A2A protocol types exactly as specified
- Ensure 100% compliance with A2A specification
- Single source of truth for protocol types
- No business logic - only type definitions

## Module Organization

### Core Types
- `task.rs` - Task lifecycle types
  - `Task` - Main task entity with id, status, history
  - `TaskStatus` - Current task state with optional message
  - `TaskState` - Enum of possible states (Submitted, Working, Completed, etc.)

### Communication Types
- `message.rs` - Message and part types
  - `Message` - Communication between agents/users
  - `Part` - Content parts (Text, Data, File)
  - `MessageRole` - Sender role (user/agent)

### Agent Types
- `agent.rs` - Agent metadata and capabilities
  - `AgentCard` - Complete agent description
  - `AgentCapabilities` - What the agent can do
  - `AgentSkill` - Individual agent skills
  - `AgentProvider` - Provider information

### Protocol Types
- `protocol.rs` - Request/response types
  - `A2aRequest` - All possible A2A requests
  - `MessageSendParams` - Parameters for sending messages
  - `TaskQueryParams` - Parameters for querying tasks
  - Various response types

### Supporting Types
- `auth.rs` - Authentication and security
  - `SecurityScheme` - OAuth2, API key definitions
  - `OAuth2Flows` - OAuth flow configurations

- `transport.rs` - Transport protocol definitions
  - `TransportProtocol` - JSON-RPC, gRPC, HTTP+JSON

- `artifact.rs` - Task artifacts
  - `Artifact` - Output generated by agents

## Usage

```rust
use crate::models::a2a::{Task, Message, AgentCard};

// All types are ready to serialize/deserialize
let task = Task {
    id: "task-123".to_string(),
    kind: "task".to_string(),
    // ...
};
```

## Important Notes

1. **No Business Logic**: These are pure data structures. Business logic belongs in services.

2. **A2A Spec Compliance**: All types must exactly match the A2A specification.

3. **Single Source of Truth**: Services should NEVER define their own protocol types.

4. **Serialization Ready**: All types implement `Serialize` and `Deserialize`.

5. **Field Names**: Use `#[serde(rename)]` to match A2A spec field names (e.g., `contextId`).
